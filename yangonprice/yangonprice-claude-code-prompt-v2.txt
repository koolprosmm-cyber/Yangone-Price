Build YangonPrice — a standalone Next.js 14 (App Router, TypeScript) demo web

CONTEXT
YangonPrice analyzes a single property listing for Yangon real estate buyers
and returns a BUY/WAIT/AVOID recommendation, entirely in Burmese. The
attached system prompt is final — do not rewrite its analytical structure.
Three things about it are deliberate and must not be "simplified" away:

1. It analyzes THREE separate actors (a regulating authority, any verified
   informal neighborhood arrangement, and the buyer's own decision fit)
   rather than treating the property itself as having policy, institutions,
   or governance. A single building cannot have its own governance — that's
   why the prompt is structured this way. Keep this three-part structure
   intact in both the API response and the UI.
2. It never compares a total price directly against a per-square-foot rate.
   Price comparisons must convert to matching units first. Preserve this
   logic in any backend code that touches price data, not just in the prompt
   text.
3. All output is Burmese only. Do not add English translations anywhere in
   the UI or API response.

CRITICAL — TERMINOLOGY BOUNDARY (read carefully, this is a hard requirement)
The system prompt's internal reasoning uses the analytical categories
"Policy," "Institutions," and "Governance," and reasons about three
distinct actors (a regulating authority, any informal arrangement, and the
buyer). This is correct internally and must stay exactly as written in the
prompt sent to OpenAI. However, NONE of this structure may be visible to the
user in any form:
- The API response sent to the frontend must expose only: `method_note`,
  `decision`, `property_summary`, `price_analysis`, `considerations`,
  `comparison`, `risk_assessment`, `recommendation`, `confidence`. The
  `_internal_actor_analysis` object in the prompt's schema exists so backend
  decision-rule code can check things like a title red flag — it must be
  used only inside the API route for that purpose and stripped out of (or
  never included in) the JSON actually returned to the browser.
- The frontend must never render three separate labeled sections for
  "authority," "informal arrangement," and "buyer fit," and must never show
  those words or the words Policy/Institutions/Governance/PIG³ anywhere —
  not as headings, not as tooltips, not as code comments visible via view
  source. The user-visible analysis is exactly two free-flowing paragraph
  fields (`property_summary` and `considerations`) — present them as natural
  prose, not as a rendering of any underlying category structure.
- Validate this yourself before finishing: search the built frontend code
  and the actual API response shape for these terms and confirm none leak
  through, including in the rendered DOM.

STACK
Next.js 14 App Router + TypeScript, Tailwind CSS, Supabase (Postgres) for
storage, Clerk for auth (scaffolded but not enforced — see AUTH section
below), OpenAI API (not Anthropic) for the analysis call, paid via a
standard OpenAI API key — no in-app payment processing needed for this demo.

REQUIRED FEATURES

1. Layout (exact specification — two boxes only)
   - The entire UI is exactly two boxes side by side, matching this
     structure: BOX 1 (left) contains every input field — township,
     property type, asking price, land area, free-text details, buyer's
     stated goal, and the submit button. BOX 2 (right) contains every piece
     of output — decision pill, the property_summary paragraph, the price
     comparison data, comparison stats, the considerations/risk text, the
     recommendation, and the confidence score. Do not split outputs across
     more than one box, and do not add extra boxes/cards for sub-sections
     of the analysis beyond what's listed.
   - Desktop/laptop (viewport ≥ ~880px): the two boxes sit side by side,
     input box narrower (roughly 1 : 1.3 width ratio), output box wider.
   - Mobile/phone (viewport < ~880px): the two boxes stack vertically,
     input box on top, output box below.
   - Implement this with a real CSS breakpoint, test it logically against
     both a typical laptop width (1440px) and a typical phone width

2. User-facing analysis flow

     where the user pastes the entire property listing as free text

     like a real Yangon property listing). Do NOT add separate fields for
     township, price, area, etc. — there is only one box, and the user

     short hint/placeholder text below the textarea is fine, and a single
     submit button.

     OpenAI API (use the Chat Completions or Responses API with JSON mode /
     structured output enforced). The prompt itself instructs the model to
     extract structured details (location, price, area, etc.) from the
     free text as its first step (see "STEP 0" in the attached system
     prompt) — do not try to pre-parse or regex the input yourself in the
     API route; let the model do the extraction as instructed.
   - The API route returns the JSON response defined in the prompt's
     schema, remapped to the neutral field names described above.
   - CRITICAL SECURITY REQUIREMENT, regardless of which LLM provider is
     used: the OpenAI API key must only ever be read from a server-side
     environment variable inside an API route or server action. It must
     never be imported, referenced, or reachable from any file marked
     `'use client'`, and never sent to or exposed in the browser in any
     form. Build this correctly from the start — do not call the OpenAI
     API directly from a client component under any circumstance.
   - A results view inside box 2 (the right-hand box) rendering, in this
